# Phase 01: Code Review Report

**Reviewed:** 2026-04-12
**Depth:** standard
**Files Reviewed:** 25
**Status:** issues_found

## Summary

The foundation data pipeline is well-structured with good patterns: `Promise.allSettled` for partial failure tolerance, epsilon-guarded floating-point comparisons in the quant engine, proper input validation in the Fear & Greed client, and generic error messages to avoid leaking internal details. The quant signal logic has thorough test coverage with boundary cases.

Key concerns are: a truthiness bug in EPS revision derivation that mishandles zero values, a color-styling bug for zero percent changes, duplicated utility code across hooks, and an unused type definition.

## Warnings

### WR-01: `deriveRevision` treats EPS estimate of 0 as missing data

**File:** `src/lib/yahoo-finance/quotes.ts:31`
**Issue:** The condition `!currentQ?.earningsEstimate?.avg` uses JavaScript truthiness, which evaluates to `true` when `avg` is `0`. This means a valid EPS estimate of $0.00 (common for breakeven companies like early-stage firms) would incorrectly return `"NEUTRAL"` instead of computing the actual revision direction.
**Fix:**
```typescript
if (currentQ?.earningsEstimate?.avg == null || nextQ?.earningsEstimate?.avg == null) return "NEUTRAL";
```

### WR-02: Zero changePercent renders with red text instead of green

**File:** `src/app/page.tsx:50-51`
**Issue:** The conditional `item.changePercent && item.changePercent >= 0` evaluates to `false` when `changePercent` is exactly `0` (because `0` is falsy in JS). This causes a 0.00% change to display in red (`text-red-600`) instead of green, which is misleading for a financial dashboard.
**Fix:**
```tsx
className={
  item.changePercent != null && item.changePercent >= 0
    ? "text-green-600"
    : "text-red-600"
}
```

### WR-03: Duplicated `getStockRefetchInterval` function across two hooks

**File:** `src/hooks/use-watchlist.ts:4-22` and `src/hooks/use-monitoring.ts:4-22`
**Issue:** The `getStockRefetchInterval` function is copy-pasted identically in both files. If market hour logic needs to change (e.g., adding holiday detection), it must be updated in two places, risking divergence and inconsistent polling behavior.
**Fix:** Extract to a shared utility:
```typescript
// src/lib/market-hours.ts
export function getStockRefetchInterval(): number { ... }
```
Then import from both hooks.

### WR-04: `YFQuoteResult` type is defined but never imported or used

**File:** `src/lib/yahoo-finance/types.ts:1-9`
**Issue:** The `YFQuoteResult` interface is defined but never referenced anywhere in the codebase. The API route handlers and `quotes.ts` use the raw yahoo-finance2 types directly. Additionally, this type is missing `regularMarketChange` which is accessed in `src/app/api/market-data/route.ts:48`. If this type was intended to be the contract for Yahoo Finance responses, it is both unused and incomplete.
**Fix:** Either remove the dead type, or apply it consistently as the return type from `yahooFinance.quote()` calls (adding the missing `regularMarketChange` field).

## Info

### IN-01: Layout metadata still uses Next.js boilerplate

**File:** `src/app/layout.tsx:17-18`
**Issue:** The page title is "Create Next App" and description is "Generated by create next app". For a financial dashboard, this should reflect the project identity.
**Fix:**
```typescript
export const metadata: Metadata = {
  title: "Alpha Engine - Quant Dashboard",
  description: "Real-time financial dashboard with quant-based action signals",
};
```

### IN-02: `FEAR_GREED_LABELS` constant defined but never used

**File:** `src/lib/constants.ts:42-48`
**Issue:** The `FEAR_GREED_LABELS` record is defined but never imported anywhere. The `fear-greed/client.ts` uses its own inline `getLabel` function with hardcoded ranges instead.
**Fix:** Either use the constant in `fear-greed/client.ts` (replacing the inline function) or remove the unused constant.

### IN-03: `console.error` calls used for server-side logging

**File:** Multiple API routes and `src/lib/fear-greed/client.ts`
**Issue:** All server-side error logging uses `console.error`. While functional, this provides no structured logging (no log levels, no request correlation, no JSON format for log aggregation). Acceptable for an MVP/Phase 1, but should be upgraded before production use.
**Fix:** Consider a lightweight structured logger (e.g., `pino`) when the project matures beyond MVP.
